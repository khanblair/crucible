Glossary

Genome
A full strategy structure: one choice of entry signal (ema_pullback, breakout, mean_reversion) plus one choice of exit style (atr_trail_half, fixed_r_multiple). Every other rule — session filter, candle quality, RSI band, 1h trend — is fixed and shared by every genome. See src/genome.py.
Regime
One of four market conditions — trending, ranging, high volatility, low volatility — classified daily from ADX and ATR percentile. Each has its own parameter file; in v1 they all share one genome. See src/regime.py.
Champion Zero
The original, hand-set day-one parameters and genome, frozen forever in config/champion_zero.json. Every optimization run replays it alongside the current system — if the evolved system can't beat this simple baseline, something is wrong with the whole premise, not the tuning.
Evaluator / Gates
The deterministic checks a candidate must pass to be accepted: meaningful improvement (10% for parameters, 20% for a genome), a 30-trade floor, a drawdown ceiling, a win-rate floor, and a consistency check against overfitting. One failure rejects the whole candidate. See src/evaluate.py.
Cooldown
No parameter change may be accepted within 5 trading days of the previous one — prevents churn and gives every change time to prove itself on new data.
Losing streak + stuck optimizer
The trigger for genome evolution: the last 5 optimization runs all show the active strategy losing money out-of-sample, AND the parameter search itself has stopped finding real improvement. Both conditions, not either — this means the logic is broken, not just unlucky.
Screen-then-refine funnel
How genome evolution searches: every candidate structure gets one cheap backtest first (screen), then only the top few get a full Optuna parameter search (refine). Cheap before expensive, always.
Auto-PR / 24-hour auto-merge
The one place automation doesn't commit directly: a winning genome opens a labeled pull request with full evidence. A human has 24 hours to review it (reminder at ~23.5h); otherwise it merges on its own, provided CI is green.
Paper-forward log
Daily hypothetical signals recorded against live candles — no real orders, ever. This is how the system checks whether its own backtest still predicts reality.
Live-vs-backtest divergence
Monthly check comparing the forward log's real results against what the backtest predicted for the same parameters. Two consecutive alerts pause new parameter changes until a human reviews the execution assumptions.
Phase 0
The one-time proof that the base strategy has any edge at all, before the optimizer is ever allowed to run on it. An optimizer cannot create an edge that isn't there — it can only find the least-losing version of a bad idea.
Intrabar ordering
When a stop and a target both fall inside one candle, which was touched first? Resolved from a committed lookup table built offline from real tick data; with no record, the engine always assumes the worst case (stop hit first).

FAQ

Why doesn't the bot just commit genome changes directly, like it does with parameters?

Changing which strategy structure is running is a bigger commitment than tuning a number. The pull-request-with-a-deadline model keeps a human in the loop by default, while the 24-hour timeout stops the system from stalling on a change that already cleared every gate.

What happens if no genome beats the baseline?

Nothing — cleanly. The attempt is logged, Discord is told, the current genome stays active, and another attempt won't be tried for a full quarter even if the losing streak continues.

Does Crucible ever place real trades?

No. Everything — backtests, the paper-forward log, genome evolution — is simulation. A separate execution layer, outside this system, would be required to trade live.

Where do the actual dollar estimates on the dashboard come from?

Deterministic arithmetic (pips × a configurable position size), never the AI report writer — money figures should never be something a language model can misstate.
